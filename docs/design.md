## Problem Statement

Build and maintain a GitHub repository of ML concepts optimized for interview prep
and quick review — not a course, not a link dump, but inline explainers with an
interview angle. Content is produced via an automated pipeline (Claude/MCP locally,
GitHub Actions for scheduling) and distributed via Instagram (Reels for discovery,
carousel posts for concepts).

The gap this fills: every existing ML GitHub repo assumes you're learning from
scratch. Nothing is built for "I have 30 minutes before a technical screen and need
to reconstruct how attention mechanisms work."

## What Makes This Cool

- **Interview-contextual explainers**: every concept includes the interview angle
  (what gets asked, what trips people up, what a great answer looks like).
- **Automated weekly drops**: GitHub Actions publishes new concepts on a schedule.
  The repo runs itself.
- **One source → two outputs**: a single markdown file produces the GitHub explainer
  AND the Instagram carousel (10-slide visual summary). No double work.
- **Ruthlessly concise format**: max 3 sub-headings, cited diagrams from reliable
  open sources, no AI-generated images.

## Constraints

1. **Content style**: concise, beginner-friendly, **exactly 3 sub-headings per post**
   (The Core Idea / How It Works / Interview Angle). Fixed structure enables carousel
   slide mapping.
2. **Format**: markdown with cited images from reliable open sources (no AI images).
3. **Metadata**: central README.md as a tagged, searchable table (Fundamentals,
   Interviews, Latest Developments, etc).
4. **Automation**: local workflow via Claude/MCP for content generation; GitHub
   Actions for scheduling and publishing.
5. **Distribution**: Instagram Reels (discovery) + carousel posts (concept).
   GitHub is the deep-dive destination.

## Premises

1. The README-as-tagged-index is the highest-value first artifact. The repo is
   useless without navigation.
2. A strict content template is a prerequisite for automation. Without a template,
   each concept takes hours; with one, it takes 20 minutes.
3. Instagram is the distribution channel: Reels for discovery, carousel posts
   (10-slide) for the concept itself.
4. The Instagram visual and GitHub explainer are produced from the same markdown
   source — one file → two outputs. The pipeline must support this split.
5. GitHub Actions handles scheduling and publishing. Claude/MCP handles content
   generation locally. The split: AI writes, Actions publishes.

## Landscape Context

- Microsoft ML-For-Beginners: 12-week curriculum, 26 lessons. Too slow for quick review.
- Awesome-ML lists: link dumps with no inline content. Navigation but no substance.
- start-machine-learning (Louis Bouchard): comprehensive guide, not concise.
- GitHub agentic workflows (Feb 2026): write workflows in Markdown, AI-driven —
  directly applicable for the automation layer.
- Multi-platform markdown publishing via Actions (to DEV.to, Hashnode) is mature.
  Instagram automation via API is available for carousel/feed posts.

## Approaches Considered

### Approach A: Manual-First, Template-Driven
Write concepts manually using a strict template. Add automation later. Instagram
output is manual (Canva). Ship something useful this weekend.
- Effort: S | Risk: Low
- Best for: validating the template and content quality before automating.

### Approach B: Full Automated Pipeline (CHOSEN)
Claude/MCP generates concept drafts from a topic queue. GitHub Actions publishes
on a schedule. Single markdown source splits into GitHub post + Instagram carousel
assets via a templated generator.
- Effort: L | Risk: Med
- Best for: a system that runs itself at scale.

### Approach C: Notion-as-CMS → GitHub Sync
Write in Notion (where one concept already exists), sync to GitHub via API.
- Effort: M | Risk: Med
- Adds Notion dependency; Instagram still manual.

## Recommended Approach: Full Automated Pipeline

### System Architecture

```
PHASE 1 (local)
────────────────────────────────────────
Topic Queue (topics.yaml)
        │
        ▼
Local: Claude/MCP generates concept draft → drafts/{slug}.md
        │
        ▼
Human review (read, edit, approve)
        │
        ▼
Set status: ready in topics.yaml, push drafts/ to repo

PHASE 1 (GitHub Actions — scheduled weekly, cron: '0 9 * * 1')
────────────────────────────────────────
        ├── Find lowest-index topic with status: ready in topics.yaml
        ├── Copy drafts/{slug}.md → concepts/{slug}.md
        ├── Update README.md index table
        ├── Mark status: published in topics.yaml
        ├── Commit and push
        └── Generate carousel PNGs via Puppeteer (slide-template.html)
              → instagram/{slug}/slide-01.png ... slide-10.png
              [Manual upload to Instagram until Graph API is set up]

PHASE 2 (future — requires Meta app review)
────────────────────────────────────────
        └── Post carousel via Instagram Graph API
              → Reel teaser (15-30s, separate asset)
```

### Content Template (concept-template.md)

Visual hierarchy (from assignment learnings): **intuition → diagram/formula → explanation → code (optional)**.
For architecture concepts, a well-chosen formula or diagram communicates in one line what 20 lines of code cannot.
Code is secondary — only include it if the mechanism cannot be shown any other way.

```markdown
---
title: "{Concept Name}"
tags: [Fundamentals | Architectures | Interviews | Latest Developments]
difficulty: beginner | intermediate | advanced
date: YYYY-MM-DD
image_source: "{Source Name} — {URL}"
image_path: "assets/{slug}.png"
---

# {Concept Name}

> One-sentence intuition. What is this, in plain English?

## The Core Idea
![{Alt text}](assets/{slug}.png)
*Source: [{Source Name}]({URL})*

{150-250 words. Intuition first, then the key formula or diagram reference. Let the visual do the heavy lifting.}

## How It Works
{150-250 words. The mechanism explained via math where relevant (LaTeX inline: $formula$).
Code snippet only if math/diagram is insufficient.}

## Interview Angle
{100-150 words. What gets asked. What trips people up. What a great answer sounds like.}
```

### GitHub Actions Workflow (publish.yml)

Trigger: `schedule: cron: '0 9 * * 1'` (weekly, Monday 9am UTC — increase cadence once quality holds)

Steps:
1. Check out repo
2. Find the lowest-index entry in `topics.yaml` with `status: ready`
3. If no `ready` entry exists: exit 0 with log message "No topics ready to publish."
4. Copy `drafts/{slug}.md` → `concepts/{slug}.md`
5. Append row to README.md index table. The index table lives at end-of-file,
   allowing simple line append. Sentinel comment `<!-- CONCEPTS_TABLE_END -->` marks
   the insertion point for the append script.
6. Run Puppeteer slide renderer: `npx puppeteer@latest` against `instagram/slide-template.html`
   (requires `actions/setup-node@v4` + `npx puppeteer browsers install chrome` — adds ~2-3 min CI)
   Output: `instagram/{slug}/slide-01.png` ... `slide-10.png`
   Cache `~/.cache/puppeteer` between runs to avoid repeat chrome downloads.
7. Set `status: published` in `topics.yaml`
8. Commit and push.
   If any step 4-7 fails: exit non-zero without committing (prevents partial state).
   Exception: if step 6 (Puppeteer) fails, commit concept file but skip PNGs, log warning.

**Permissions required in workflow YAML:**
```yaml
permissions:
  contents: write
```
Uses default `GITHUB_TOKEN`. Branch protection rules must allow Actions bot commits,
or a PAT stored as `GH_PAT` secret must be used instead.

**Phase 2 only** (after Meta app review): add step to POST carousel to Instagram Graph API
using `INSTAGRAM_ACCESS_TOKEN` secret.

### Instagram Carousel Pipeline

Each concept markdown generates 10 slides:
- Slide 1: Concept title + one-line hook (bold, minimal)
- Slides 2-4: Core idea broken into 3 visual chunks
- Slides 5-7: How it works (diagrams, code snippets if relevant)
- Slides 8-9: Interview angle (common question + what a great answer includes)
- Slide 10: CTA — "Full explainer + more concepts: [GitHub link in bio]"

Rendering:
- **Tool**: Puppeteer renders `instagram/slide-template.html` → 1080x1080px PNG
- **Slide template**: `instagram/slide-template.html` (dark background #0d1117,
  font: system-ui / Geist Mono for code, white text).
- **Slide data contract**: each slide's content is injected via `window.__SLIDE_DATA__`
  before render. JSON shape per slide:
  ```json
  { "title": "string", "body": "string", "code": "string|null", "image_url": "string|null", "slide_num": 1 }
  ```
  The renderer script sets `window.__SLIDE_DATA__` and calls `puppeteer.evaluate()` to
  inject before screenshot. All 10 slides are generated in one Puppeteer session
  (loop over slide data array).
- **Resolution**: 1080x1080px (Instagram carousel requirement).
- **Puppeteer in CI**: requires `actions/setup-node@v4` + chrome install step.
  Cache restore must come *before* the install to be effective:
  ```yaml
  - uses: actions/cache@v4
    with:
      path: ~/.cache/puppeteer
      key: puppeteer-${{ runner.os }}-${{ hashFiles('package-lock.json') }}
      restore-keys: puppeteer-${{ runner.os }}-
  - run: npx puppeteer browsers install chrome
    env:
      PUPPETEER_CACHE_DIR: ~/.cache/puppeteer
  ```
- **Source diagrams**: cited from Wikimedia Commons, distill.pub, Papers With Code.
  No AI-generated images. Image URL stored in frontmatter `image_source` field.
- Word count guidelines (150-250 words per section) are **soft guidelines for manual
  review**, not enforced by CI lint.

### Topic Queue (topics.yaml structure)

```yaml
queue:
  - slug: attention-mechanism
    title: Attention Mechanism
    tags: [Architectures, Interviews]
    difficulty: intermediate
    status: ready   # ready = reviewed, cleared to publish
  - slug: batch-normalization
    title: Batch Normalization
    tags: [Fundamentals, Interviews]
    difficulty: beginner
    status: queued  # queued = not yet drafted
```

**Status lifecycle:**
- `queued` — topic is in the list, not yet drafted
- `draft` — Claude has generated `drafts/{slug}.md`, human review pending.
  Set automatically by the CLI command after writing the draft file.
- `ready` — reviewed and approved, cleared for publish by Actions.
  Set manually after human review.
- `published` — committed to `concepts/` by Actions.

**Slug convention:** `slug = title.lower().replace(' ', '-')`, e.g.
"Attention Mechanism" → `attention-mechanism`. Must match `topics.yaml` entry exactly.

Actions only picks entries with `status: ready`. Selection rule: lowest list index
(i.e. FIFO — add topics to the bottom of the list to queue them).

### Local Workflow (Claude/MCP)

#### Step 1: Capture source image (browser screenshot — bypasses hotlink protection)

curl fails against most ML image sources (Wikipedia, distill.pub, Papers With Code)
due to hotlink protection. Use the gstack browse binary to screenshot instead:

```bash
B=~/.claude/skills/gstack/browse/dist/browse  # or repo-local equivalent

# Navigate to source page and screenshot a specific element
$B goto "https://en.wikipedia.org/wiki/Transformer_(deep_learning_architecture)"
$B screenshot assets/$SLUG.png --selector ".thumb img"

# Or full-page screenshot, then crop manually
$B screenshot /tmp/$SLUG-full.png
```

For distill.pub articles, Papers With Code diagrams, or any protected source —
same pattern. The browse binary renders the page in a real browser; no hotlinks apply.

Store the captured image in `assets/{slug}.png`. Record the source URL in frontmatter.

#### Step 2: Generate concept draft

```bash
SLUG="attention-mechanism"  # match topics.yaml entry

claude "Using concept-template.md, write a concept for Attention Mechanism.
        The source image is already saved at assets/$SLUG.png.
        Exactly 3 sub-headings: The Core Idea, How It Works, Interview Angle.
        Prioritize math/formulas over code. Beginner-friendly.
        Output only the markdown content with no preamble." \
  > drafts/$SLUG.md

# Update topics.yaml: set status → draft
python3 -c "
import re, sys
txt = open('topics.yaml').read()
txt = re.sub(
  r'(slug: $SLUG\b.*?\n\s+status:) queued',
  r'\1 draft', txt, flags=re.DOTALL
)
open('topics.yaml','w').write(txt)
"
```

#### Step 3: Review and approve

```bash
# Review drafts/$SLUG.md — edit as needed
# When satisfied, mark ready:
python3 -c "
import re
txt = open('topics.yaml').read()
txt = re.sub(
  r'(slug: $SLUG\b.*?\n\s+status:) draft',
  r'\1 ready', txt, flags=re.DOTALL
)
open('topics.yaml','w').write(txt)
"

git add assets/$SLUG.png drafts/$SLUG.md topics.yaml
git push
# Actions picks it up on next Monday 9am UTC run
```

## The Assignment

Before building any automation: **write 3 concepts using the template above by hand.**
Pick attention, backprop, and one architecture (transformers or CNNs). Write them in
the exact format. If writing one concept takes more than 45 minutes, the template
needs simplification. If it takes less than 20 minutes and quality is high — automate it.

The template is the product. Don't build the pipeline until the template works.
